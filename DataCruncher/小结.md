### 1. 文件角色与包含逻辑 (C++ 层面)

| 文件类型 | 核心职责 | 包含原则 |
| --- | --- | --- |
| **`.h` (头文件)** | **声明契约**：告诉外部有哪些函数、类、结构体。 | 必须加 **宏卫语句** (`#ifndef`) 或 `#pragma once`。只包含声明所需的最小库（如 `<vector>`）。 |
| **`.cpp` (源文件)** | **实现逻辑**：具体的代码执行逻辑。 | **首行**包含对应的 `.h`。包含实现所需的重型库（如 `<algorithm>`），不暴露给外部。 |
| **`main.cpp`** | **程序入口**：组装各模块，驱动业务流程。 | 显式包含自己直接用到的所有头文件（如 `<iostream>`, `<vector>`），不靠“蹭”。 |

### 2. CMake 核心指令 (构建层面)

这是你管理项目“身体”的四把钥匙：

1. **`include_directories(include)`**
* **作用**：把 `include` 文件夹加入全局搜索路径。
* **效果**：你的代码里可以写 `#include "calc.h"` 而不是相对路径 `#include "../include/calc.h"`。


2. **`add_library(CruncherCore src/calc.cpp)`**
* **作用**：将算法实现打包成一个独立的二进制“零件”（静态库）。
* **意义**：实现了逻辑的**封装**和**复用**。


3. **`add_executable(DataApp src/main.cpp)`**
* **作用**：定义最终要运行的那个程序。


4. **`target_link_libraries(DataApp CruncherCore)`**
* **作用**：红娘指令。把“主程序”和“算法库”黏合在一起。
* **注意**：这里必须用**目标名**（Target Name），拼写错误（如写成 `directories`）会导致编译失败。



---

### 3. 开发流程与编译原理

* **编译 (Compile)**：`.cpp` $\rightarrow$ `.o` (或 `.obj`)。将源代码翻译成机器码碎片。
* **链接 (Link)**：多个 `.o` + 库文件 $\rightarrow$ 可执行文件。把碎片拼接成完整的程序。
* **增量编译**：CMake 聪明到只重新编译你改动过的那个文件，这在大项目中能节省几小时的时间。

---

### 4. 工程化最佳实践

* **接口与实现分离**：让使用者只看菜单（`.h`），不进后厨（`.cpp`）。
* **依赖最小化**：不要在头文件中包含不必要的重型库，减小编译压力。
* **源外构建 (Out-of-Source Build)**：将 `build` 文件夹与源码分开（无论是在项目内还是项目外），保持代码目录的纯净。

---

### 5. 调试总结

* **错误排查**：当看到 `Undefined symbols` 时，通常是忘了链接库（`target_link_libraries`）；当看到 `Redefinition` 时，通常是忘了宏卫语句。
* **工具包 (Kit)**：在 VS Code 中，确保选择了正确的编译器版本（如 Apple Clang++）。

### 2. 核心技术发现

在监控过程中，我们挖掘出了三个影响 C++ 程序效率的“真凶”：

* **内存颠簸 (Memory Thrashing)**：原本 `MyVector` 的动态扩容像是一个搬家公司，每次只给新家加一个房间，导致 100 万次搬家。`reserve` 预留空间让它一次到位。
* **对象重量 (Object Overhead)**：`std::stringstream` 是个“全能翻译官”，它能处理各种语言、格式和错误状态，但对于我们这种追求速度的“数字工厂”来说，它太重了。
* **I/O 瓶颈**：实验数据证明，无论计算多快，数据从硬盘进入程序的这一关永远是最慢的。读取时间 (0.37s) 依然是计算时间 (0.007s) 的 **50 倍** 以上。

---

### 3. 架构设计的权衡 (Trade-offs)

在优化过程中，我们经历了一次重要的决策转变：

* **理想 vs. 现实**：我们本想用最现代化的 `std::from_chars`，但遇到了 macOS 编译器的实现缺失。
* **工程方案**：我们果断切换到 `std::strtod`。虽然它是个 C 风格的函数，但在保证了**百万级数据 0.37s** 的高性能的同时，也兼顾了**跨平台兼容性**。这体现了工程开发中“不选最贵，选最对”的原则。

目前你的 `DataCruncher` 已经达到了一流的工业水平：

1. **高吞吐量**：支持每秒约 270 万条浮点数的解析。
2. **低延迟**：排序与统计计算（均值/方差）控制在 0.15s 以内，适合准实时处理。

## 1. `resize(size_t new_size, const T& value = T())`

`resize` 的本质是“只调整有效对象的数量，不轻易改变底层地皮（内存）”。它在面对三种不同的尺寸变化时，底层的博弈截然不同：

* **情况 A：尺寸变大且超过当前容量（`new_size > _capacity`）**
1. 触发倍增扩容（调用 `reserve`），向系统申请一块全新的大内存。
2. 利用 `std::move_if_noexcept` 将旧对象高效“搬家”到新大楼，并销毁老家对象。
3. 利用 Placement New 在剩余的新格子中，通过**拷贝构造**将参数 `value`（若未传参则是默认构造的 `T()`）填充进去。


* **情况 B：尺寸变大但未超过当前容量（`_size < new_size <= _capacity`）**
* **极速性能**：不需要申请任何内存！直接在原本就持有的空闲格子上，用 Placement New 把新对象孵化出来。


* **情况 C：尺寸变小（截断，`new_size < _size`）**
* **拆房不退地策略**：**只销毁对象，不释放内存**。精准地对尾部多出来的对象手动调用析构函数 `~T()`，清空格子，并将 `_size` 修改为 `new_size`。但 `_capacity` 保持不变，留作后续高频复用的“高速缓存”。



---

## 2. `shrink_to_fit()`

`shrink_to_fit` 是对“拆房不退地”策略的完美补充，它是给容器**显式瘦身**的剪刀。

* **核心动作**：
1. 当发现有闲置地皮（`_capacity > _size`）时，精准申请一块**刚刚好**能容纳当前 `_size` 个对象的新内存。
2. 将当前容器内仅存的有效对象**移动构造**（或拷贝）到新内存中。
3. 顺手析构掉旧内存上的有效对象。
4. 调用 `::operator delete(data)` 将原本庞大臃肿的旧地皮**彻底归还给操作系统**。


* **工业价值**：日常不自动触发，把“何时归还大内存以节省系统资源”的最高决定权完全交回给开发者。

---

## 3. `emplace_back(Args&&... args)`

`emplace_back` 是现代 C++ 泛型编程的性能神技，它的终极目标是**彻底消灭容器插入时的“临时对象”与“搬迁开销”**。

* **技术双子星**：
* **可变参数模板（`Args&&...`）**：像万能口袋一样，可以接收任意数量、任意类型的构造参数。
* **完美转发（`std::forward`）**：原封不动地保持参数的左值/右值属性，一路免损耗地护送到最底层。


* **底层行为（对比 `push_back`）**：
* `push_back(Student(1, "Alex"))`：先在外面（栈上）造个临时房子 $\rightarrow$ 拷贝/移动进 Vector 内部 $\rightarrow$ 析构外面房子的空壳（三部曲）。
* `emplace_back(1, "Alex")`：不接收对象，只接收参数。直接在 Vector 尾部的原始内存格子上执行 `new (data + _size) T(std::forward<Args>(args)...)`。**对象在整个宇宙中只被构造了一次，原地孵化，一步到位。**
